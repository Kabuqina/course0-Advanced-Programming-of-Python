# Lesson 16 讲义 · 综合项目（雏形）

综合项目的目标不是学新语法，而是**把已学机制组织成一个真实可用的小工具**，并保持
可运行、可测试、可讲解。

## 以选题 A 为例的分层设计
1. **数据模型**：用 `dataclass` 表示一次检查结果（lesson 名 + 缺失文件列表）。
2. **纯逻辑**：`check_lesson(dir)` 返回缺失文件；`check_course(root)` 遍历所有 lesson。
   把“扫描目录”与“格式化输出”分开，逻辑纯、易测。
3. **遍历**：用生成器惰性遍历子目录，避免一次性构造大列表（Lesson 03）。
4. **输出**：`format_report(report, fmt)` 支持 markdown / plain，UI 与逻辑解耦。
5. **CLI**：`argparse` 暴露 `--root` / `--format`；`python -m ...` 运行（Lesson 12 预告）。
6. **可观测**：用 `logging`（Lesson 08 预告）而非 `print` 记录进度。
7. **测试**：用 `tmp_path` 造几个假 lesson 目录，断言缺失检测正确（Lesson 11）。

## 讲解要点
- 先让工具**能跑**（哪怕功能少），再逐步加特性——避免一开始就追求大而全。
- 把“做什么”（业务规则：需要哪些文件）与“怎么呈现”（markdown 表格）分离。
- 每加一个功能，先补一个测试。

## 从雏形到完整
- 给 A 增加：忽略非课程目录、检查 `metadata.yml` 字段是否齐全、输出 JSON、退出码非零表示有缺失（便于 CI）。
- 选 B/C 的同学，可参照这套“数据模型 → 纯逻辑 → 遍历 → 输出 → CLI → 测试”的骨架落地。
